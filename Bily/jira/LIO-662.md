# LIO-662 · API - Refactor - Agregar caché Redis al recurso de opciones de precio/vendedores para un item

[LIO-662 en Jira](https://bluinc.atlassian.net/browse/LIO-662)

## Descripción

Agregar caché con Redis al endpoint `POST /v4/resellersByItemId` para mejorar el tiempo de respuesta y reducir consultas repetidas a la base de datos.

**Contexto**  
El recurso recibe una lista de `itemId` junto con parámetros como `offset`, `limit` y `order`, y devuelve los resellers disponibles por producto. Como esta información puede ser consultada repetidamente, se debe cachear la respuesta durante 1 hora.

**Alcance**  
Implementar Redis sobre el recurso `resellersByItemId`.

La clave de caché debe contemplar:

- `payload`
- `offset`
- `limit`
- `order`

**Comportamiento esperado**  
Si existe una respuesta vigente en Redis, el endpoint debe devolver la información desde caché sin consultar la base de datos.

Si no existe caché, el endpoint debe consultar normalmente, devolver la respuesta y guardar el resultado en Redis con un TTL de 1 hora.

No se deben cachear respuestas con error o excepciones.

Las respuestas vacías exitosas sí pueden cachearse.

El formato actual de respuesta no debe modificarse.

**Criterios de aceptación**

1. Al consultar el endpoint con los mismos parámetros, la segunda respuesta debe salir desde Redis.
2. La caché debe expirar automáticamente luego de 1 hora.
3. Cambios en `payload`, `offset`, `limit` u `order` deben generar una clave distinta.
4. Si ocurre un error en la consulta, esa respuesta no debe guardarse en Redis.
5. La respuesta del endpoint debe mantener la misma estructura actual.
