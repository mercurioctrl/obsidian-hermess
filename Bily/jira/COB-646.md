# COB-646 · API COB fix cobro capital cliente log

[COB-646 en Jira](https://bluinc.atlassian.net/browse/COB-646)

## Descripción

**Problema**

  
Al cobrar un préstamo de capital

```
POST /payCapitalDebt
```

, el movimiento de caja no mostraba el cliente asociado.  
Al consultar

```
 GET /box/{boxId}
```

, clientId y clientName devolvían null o vacío.

Causa raíz  
MC_LOG_OPERACIONES se insertaba con ID_REFERENCIA = 0. La consulta de caja busca el cliente mediante el JOIN  
MC_LOG_OPERACIONES → MC_CCORRIENTES_MOVIMIENTOS (ID_REFERENCIA) → clientes, que nunca coincide porque no existe un movimiento de cuenta  
corriente con ID_CCMOVIMIENTO = 0.

**Solución**

Base de datos  
Se agregó la columna ID_PRESTAMOS_CAPITAL INT NULL a MC_LOG_OPERACIONES para vincular el log de caja directamente a MC_PRESTAMOS_CAPITAL,  
evitando la reutilización de ID_REFERENCIA que causaba colisiones entre las secuencias IDENTITY independientes de ambas tablas.

PayCapitalDebtRepository

- insertPaymentRecord() ahora retorna int con el ID_PRESTAMO generado (lastInsertId()).
- registerPaymentLog() recibe ese ID y lo escribe en la nueva columna ID_PRESTAMOS_CAPITAL. ID_REFERENCIA se mantiene en 0.

PayCapitalDebtService  
Se reordenaron las llamadas al repositorio: primero se inserta el registro en MC_PRESTAMOS_CAPITAL para obtener el ID,  
luego se registra el log de caja con ese ID y finalmente se actualiza el saldo.

BoxMovementRepository — método select()  
Se agregaron dos LEFT JOINs:

- MC_PRESTAMOS_CAPITAL PCAP ON PCAP.ID_PRESTAMO = LOG.ID_PRESTAMOS_CAPITAL
- clientes clientPC ON clientPC.ID_CLIENTE = PCAP.ID_CLIENTE

Los campos clientId y clientName usan COALESCE para resolver el cliente por la ruta existente (cuenta corriente) y la nueva (préstamo de capital), sin afectar otros tipos de movimiento.
