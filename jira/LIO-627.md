# LIO-627 · API - Feat - Migrar recurso para guardar en favoritos desde la ficha del producto

[LIO-627 en Jira](https://bluinc.atlassian.net/browse/LIO-627)

## Descripción

Como equipo de backend, queremos implementar el recurso `/favoritos` en la API v4 (Laravel) replicando el comportamiento de la API v3 (legado), para poder deprecar la dependencia del cliente en la API legada y unificar el contrato de datos bajo el nuevo stack.

---

## Contexto

La API v3 expone un recurso completo de favoritos. La API v4 **no tiene nada implementado** para favoritos. Esta historia cubre la migración completa del recurso, con especial énfasis en los endpoints de ítem individual (`/favoritos/item`) que son el foco principal del request.

No hay cambios de base de datos: la tabla ya existe y se reutiliza tal cual.

---

## Tabla involucrada

**Base de datos:** `[LO]` **Tabla:** `[LO].[dbo].[favoritos]`

ColumnaTipoDescripción`productoId`INTID del producto guardado como favorito`usuarioId`INTID del usuario propietario del favorito

> No hay columna de fecha ni de ID propio de la fila expuestos en la API v3. Si la tabla los tiene, no forman parte del contrato público.

La tabla **no tiene índice único documentado**, pero la lógica de v3 previene duplicados a nivel aplicación (consulta antes de insertar).

---

## Endpoints a implementar

Todos los endpoints requieren autenticación JWT (`Authorization: Bearer <token>`). El `usuarioId` **nunca viene del body**: se extrae del token autenticado.

### 1. `GET /v4/favoritos`

Devuelve la lista completa de productos guardados como favoritos por el usuario autenticado.

**Request:**

```
GET /v4/favoritos
Authorization: Bearer <token>
```

Sin body.

**Response exitosa **`200 OK`**:**

```
[
  {
    "id": 123,
    "titulo": "Nombre del producto",
    "precio": 1500.50,
    "imagen": "url-imagen",
    ...
  }
]
```

> Devuelve los datos completos de cada producto (los mismos campos que el endpoint de ficha de producto). Si no hay favoritos, devuelve array vacío `[]`.

---

### 2. `POST /v4/favoritos`

Agrega múltiples productos a favoritos en una sola llamada. Omite silenciosamente los que ya están guardados (no duplica).

**Request:**

```
POST /v4/favoritos
Authorization: Bearer <token>
Content-Type: application/json
​
{
  "productos": [
    { "id": 123 },
    { "id": 456 }
  ]
}
```

**Response exitosa **`200 OK`**:** Sin body (o body vacío). La v3 no devuelve nada en este caso.

**Errores:**

- `422` si falta el campo `productos` en el body.
- `500` si falla la inserción.

---

### 3. `POST /v4/favoritos/item` ⭐ (foco del request)

Agrega **un único producto** a los favoritos del usuario autenticado. Si el producto ya está en favoritos, devuelve error (no duplica).

**Request:**

```
POST /v4/favoritos/item
Authorization: Bearer <token>
Content-Type: application/json
​
{
  "id": 123
}
```

**Response exitosa **`200 OK`**:**

```
{
  "status": "OK",
  "item": {
    "id": 123,
    "titulo": "Nombre del producto",
    "precio": 1500.50,
    "imagen": "url-imagen",
    ...
  }
}
```

> Devuelve el producto completo dentro de `item`, igual que lo haría el endpoint de ficha.

**Errores:**

- `422` si falta el campo `id` en el body.
- `409 Conflict` si el producto ya estaba en favoritos. *(En v3 devolvía 500; en v4 usar 409 es más correcto semánticamente — coordinar con frontend si cambia el código).*
- `500` si falla la inserción por otro motivo.

---

### 4. `DELETE /v4/favoritos/item/{productoId}` ⭐ (foco del request)

Quita **un único producto** de los favoritos del usuario autenticado.

**Request:**

```
DELETE /v4/favoritos/item/{productoId}
Authorization: Bearer <token>
```

Sin body. El `productoId` va en la URL como parámetro de ruta.

**Response exitosa **`200 OK`**:**

```
{
  "status": "OK"
}
```

**Errores:**

- `400` si no se provee `productoId`.
- `500` si falla el DELETE.

---

### 5. `DELETE /v4/favoritos`

Vacía **todos** los favoritos del usuario autenticado.

**Request:**

```
DELETE /v4/favoritos
Authorization: Bearer <token>
```

Sin body.

**Response exitosa **`200 OK`**:**

```
{
  "status": "OK"
}
```

**Errores:**

- `500` si falla el DELETE masivo.

## Criterios de aceptación

- `POST /v4/favoritos/item` agrega un producto y devuelve sus datos completos
- `POST /v4/favoritos/item` devuelve error si el producto ya estaba en favoritos (no duplica)
- `DELETE /v4/favoritos/item/{productoId}` elimina el producto de los favoritos del usuario
- `DELETE /v4/favoritos/item/{productoId}` solo elimina el favorito del usuario autenticado (no afecta otros usuarios)
- `GET /v4/favoritos` devuelve array vacío si el usuario no tiene favoritos
- `GET /v4/favoritos` devuelve los datos completos de los productos (no solo IDs)
- `POST /v4/favoritos` no duplica productos que ya estaban guardados
- `DELETE /v4/favoritos` vacía únicamente los favoritos del usuario autenticado
- Todos los endpoints rechazan requests sin token válido con `401`
- El `usuarioId` siempre se extrae del token JWT, nunca del body ni de query params

## Notas técnicas

- **No requiere migración de base de datos.** La tabla `[LO].[dbo].[favoritos]` ya existe y tiene los datos de producción.
- **Prevención de duplicados a nivel aplicación:** La v3 consulta antes de insertar. En v4 se puede conservar este enfoque o agregar una constraint UNIQUE en DB — consultar con el tech lead.
- **Datos del producto:** El `GET /favoritos` y el `POST /favoritos/item` devuelven el producto completo. Reutilizar la query/método de `ItemRepository` que ya existe en el proyecto para obtener datos por ID, en lugar de hacer una query nueva.
- **Código de error en duplicado:** La v3 lanzaba 500 para producto ya en favoritos. En v4 usar 409 es más semánticamente correcto; coordinar con el equipo frontend antes de decidir.
- **Sin caché:** Este recurso es personal por usuario. No aplicar caché Redis.
- **Sin paginación:** La v3 no pagina favoritos. Mantener el mismo comportamiento por ahora.
