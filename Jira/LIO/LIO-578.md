## Descripción

# Verificación de teléfono real por SMS en el flujo de verificación del vendedor

Se requiere que el vendedor verifique su número de teléfono real mediante un código OTP enviado por SMS (Twilio) antes de poder enviar el formulario de verificación. El objetivo es garantizar que el número de teléfono ingresado en `real_phone` sea válido y pertenezca al vendedor.

---

## Contexto del negocio

El formulario de verificación del vendedor (`/mi-cuenta/configuracion/verificar-vendedor`) permite que un vendedor envíe sus datos personales (nombre real, domicilio real y teléfono real) junto con fotos del DNI para ser validado por el equipo de Libre Opción.

Actualmente, el campo `real_phone` se guarda tal como lo ingresa el vendedor, sin ninguna validación de que el número sea real o accesible. Esto permite que se ingresen teléfonos ficticios o erróneos.

Con esta feature, antes de poder enviar el formulario, el vendedor debe verificar su número de teléfono ingresando un código de 6 dígitos recibido por SMS. Solo después de verificar el número el botón "Verificar vendedor" se habilita.

**Reglas de negocio clave:**

- El código OTP tiene una validez de 10 minutos desde su generación.


- Si el vendedor cambia el número de teléfono después de haberlo verificado, la verificación se invalida y debe repetirse.


- Si ya existe un `real_phone` guardado en el servidor para ese vendedor, el teléfono arranca como pre-verificado (no requiere OTP al cargar la página, solo si lo modifica).


- Se permite un máximo de 3 reenvíos de código por usuario por sesión de 60 minutos.


- Se permite un máximo de 5 intentos de verificación por código; si se agotan, el código se invalida y hay que reenviar.

## Endpoints nuevos

### 1. Enviar código OTP

```
POST /phone-verification/send
```

Endpoint nuevo bajo `token.auth`. Genera un código OTP de 6 dígitos, y envía el SMS al número indicado vía Twilio.

#### Response esperada — 200 OK

```
}
```

## Cambios en el endpoint existente

```
POST /vendor-verification
```

Extender la validación existente para que, antes de guardar, verifique que el teléfono haya sido verificado por OTP. Específicamente:

- Si no existe o el `phone` no coincide con `real_phone` del form → responder `400` con mensaje `"El teléfono no fue verificado."`.


- Si existe y coincide → continuar con el flujo normal de guardado.

## Cambios en el frontend (`verificar-vendedor.vue`)

El campo `real_phone` pasa por tres estados visuales:

**Estado 1 — Editable (sin verificar):**

- Input habilitado para tipear el teléfono.


- Aparece el botón "Enviar código" una vez que el número tiene formato válido (misma regex actual).


- El botón "Verificar vendedor" está deshabilitado.



**Estado 2 — Código enviado (pendiente):**

- Input del teléfono en `readonly` con el número ingresado.


- Aparece un nuevo input para el código OTP de 6 dígitos.


- Botón "Verificar código".


- Link "Reenviar código" con cooldown visual de 60 segundos.


- El botón "Verificar vendedor" sigue deshabilitado.



**Estado 3 — Teléfono verificado:**

- Input del teléfono en `readonly` con badge "Verificado ✓" en verde.


- Los inputs del OTP se ocultan.


- El botón "Verificar vendedor" se habilita (junto con el resto de las condiciones actuales: nombre, domicilio, DNI frente y dorso).


- Si el usuario hace click en el teléfono para editarlo, se vuelve al Estado 1 y se limpia la verificación.



**Caso de teléfono pre-existente (vendor ya verificado):**

- Si `getVendorVerification()` devuelve un `real_phone` cargado, el campo arranca directamente en Estado 3 (`phoneVerified = true`).


- Si el usuario edita el número, vuelve al Estado 1.



**Variables de estado a agregar:**

```
phoneVerified: false,
phoneOtpSent: false,
otpCode: '',
sendingOtp: false,
verifyingOtp: false,
resendCooldown: 0,   // segundos restantes para poder reenviar
```

**Condición **`showDisabledSubmit`** actualizada:** Agregar `!this.phoneVerified` a las condiciones existentes.

## Criterios de aceptación

- El botón "Verificar vendedor" permanece deshabilitado hasta que el teléfono haya sido verificado por OTP.


- Un código OTP expira a los 10 minutos de su generación; pasado ese tiempo, intentar verificarlo devuelve un error `400`.


- Si el vendedor cambia el número luego de haberlo verificado, el estado de verificación se limpia y debe repetir el proceso de OTP.


- Si ya existe un `real_phone` guardado en el servidor para el vendedor, el formulario carga con el teléfono pre-verificado y el botón "Verificar vendedor" habilitado (sin necesidad de nuevo OTP, a menos que cambie el número).


- Tras 5 intentos fallidos de verificación, el código se invalida y el usuario debe solicitar uno nuevo.


- No se permite solicitar más de 3 códigos por usuario en un período de 60 minutos; el endpoint `/send` responde `429` si se supera ese límite.


- El endpoint `POST /vendor-verification` existente rechaza el envío con `400` si el teléfono enviado no coincide con el `phone_verified:{userId}` 


- El campo `real_phone` del formulario queda en `readonly` mientras hay un OTP pendiente o el teléfono está verificado; se puede editar solo si el usuario hace click explícito para cambiar el número.
