## Descripción

Refactorizaremos el recurso

```
POST /aleph/compatibility
```

para parametrizar en el `.env` el parametro `COMPATIBILITY_SYS_ROLE_CONTENT`

ya que buscamos evitar respuestas con voz propia, probaremos con este valor inicial

```
Respondés como un sistema automático de validación de compatibilidad de hardware. Tu salida debe ser siempre en formato JSON con los siguientes campos:

- "compatibility": un booleano que indique si todos los productos del listado son compatibles entre sí (true o false).
- "message": un texto breve (máximo 2 oraciones) en tono profesional, informativo y estrictamente neutral. Debe comenzar con "El análisis con IA detecta que...".

No uses frases de cortesía ni respondas como si te hubieran hecho una pregunta.

```

Para evitar respuesta de este tipo que empiezan como contestando una pregunta

```json
{
    "respuesta": "No, estos productos no son compatibles entre sí. El procesador Intel Core i3 10105 requiere un socket LGA1200, pero la placa madre Gigabyte A520M K V2 1.1 tiene un socket AM4, que es para procesadores AMD, no Intel."
}
```

Lo que buscamos es obtener este tipo de respuesta

```
{
  "compatibility": false,
  "message": "El análisis con IA detecta que los productos no son compatibles entre sí. El procesador Intel Core i3 10105 utiliza socket LGA1200, pero la placa madre Gigabyte A520M K V2 tiene socket AM4 para procesadores AMD."
}

```

### 📌 Recomendaciones

- **Temperature**: bajo (ej: 0.2–0.3) para mayor consistencia.


- **Validá que siempre devuelva JSON**: agregá lógica de fallback por si responde mal.



Estos esquemas son ostentativos, así que todas las sugerencias son bienvenidas y los cambios deben aplicarse a la historia original.
