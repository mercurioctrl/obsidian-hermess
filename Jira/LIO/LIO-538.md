## Descripción

Migrar el recurso de productos similares desde la API Legacy hacia la API v4, manteniendo la funcionalidad actual pero promoviendo una **reimplementación consciente y mejorada**.

El objetivo no es replicar el existente tal cual esta, sino **rediseñar su lógica con criterio técnico**, permitiendo que el desarrollador analice el comportamiento actual, proponga mejoras en el algoritmo de similitud, optimice performance y defina un contrato más robusto y mantenible para el futuro.

Se espera planificación técnica, validación de supuestos y una propuesta que eleve la calidad del resultado final en términos de relevancia, eficiencia y escalabilidad.


Hoy existe en la API Legacy el endpoint:

- **Legacy:** `POST {API_LEGACY}/productos/ficha/similares`


- Ejemplo de payload:



```
{
  "id": 442524,
  "categoria": { "id": 34, "nombre": "Soportes", "uri": "soporte" },
  "marca": { "id": 3167, "nombre": "INTELAID", "img": "7e2f27ceec964247628431cc7a2d7aae.png" }
}

```

Se necesita **migrar la funcionalidad** a la **API nueva v4 (Laravel)**, pero **no como “copiar y pegar”**: la idea es que v4 tenga **su propia implementación**, mejor definida, más eficiente, y con un resultado más consistente para el usuario.

Migrar el recurso de productos similares desde la API Legacy a la API v4 mediante una reimplementación planificada, evitando la copia directa. Se busca que el desarrollador analice el comportamiento actual y proponga una solución mejorada en lógica, performance y mantenibilidad.

---

## La “gracia” del repositorio de similares (por qué existe)

Este repositorio no es “otro listado más”: es el motor que alimenta el bloque **“Productos similares”** en la **ficha de producto**.

### Para qué sirve (objetivo real)

Cuando un usuario entra a la ficha de un producto:

- si **no compra ese producto**, queremos **evitar que se vaya**;


- si el producto está **caro / sin stock / no le convence la marca**, queremos ofrecerle alternativas cercanas;


- si el usuario está explorando, queremos darle un “siguiente paso” sin obligarlo a volver al buscador.



En resumen: **retención, conversión y navegación** dentro del catálogo.

### Dónde se usa (ubicación funcional)

- **Frontend / Ficha de producto**: sección “Similares”.


- Se dispara típicamente **después de cargar el producto principal** (ya tenés `id`, `categoria`, `marca`).


- No debería bloquear la carga principal: idealmente es un request “secundario” (optimizable con caché).



---

## Problema típico del Legacy (lo que hay que mejorar en v4)

El endpoint legacy recibe `id + categoria + marca`. Eso sugiere dos riesgos:

- **Confianza en datos que manda el cliente**
Si el cliente te manda categoría/marca “equivocadas” (por bug o manipulación), el algoritmo devuelve basura.


- **Algoritmo pobre o rígido**
Muchas implementaciones de “similares” terminan siendo:

- “mismo categoryId y listo”


- o “misma marca y listo”
Resultado: recomendaciones flojas, repetidas, o irrelevantes.





La v4 debería:

- **derivar la categoría y marca reales desde DB** a partir del `productId`,


- y recién después decidir el set final.

# Requerimiento

**Como** usuario en la ficha de un producto,**quiero** ver productos similares relevantes, **para** comparar alternativas y seguir navegando sin volver al buscador.

# Respuesta del endpoint (contrato)

Devolver un payload compacto, orientado a UI:

```
{
  "productId": 442524,
  "limit": 12,
  "items": [
    {
      "id": 123,
      "title": "Soporte XYZ",
      "price": 9999.99,
      "currency": "ARS",
      "brand": { "id": 3167, "name": "INTELAID" },
      "category": { "id": 34, "name": "Soportes", "uri": "soporte" },
      "mainImage": "https://...",
      "stock": 5,
      "uri": "/producto/soporte-xyz",
      "badges": ["same_brand", "close_price"]
    }
  ]
}

```

Notas:

- `badges` ayuda a explicar por qué aparece (“misma marca”, “precio similar”), mejora percepción de calidad.


- Evitar traer campos pesados que no usa el carrusel.



---

# Reglas / Constraints

- No debe incluir el mismo producto (`exclude current id`).


- Debe respetar visibilidad (ocultos, borrados lógicos, etc.).


- Debe soportar `limit` y “includeOutOfStock”.



---

# Performance y caché

- Cache por `productId + filtros` (ej. 5–30 min).


- Invalidación: cuando cambia categoría/marca/estado/stock (si tenés eventos), o TTL si no.


- Query optimizada: traer solo columnas necesarias.
