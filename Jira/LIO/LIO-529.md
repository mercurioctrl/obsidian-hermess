## Descripción

Endpoint existente. Se amplía el objeto `user` para incluir el estado de verificación del vendedor.

---

## Endpoint

Verb

Route

GET

`/auth/user`

---

## Cambios funcionales

El endpoint ya devuelve el objeto `user` basado en `UserDto`.
Se agregan dos propiedades nuevas provenientes de la tabla `vendor_verification`.

---

### Nuevos campos

Field

Type

Origen

vendorVerified

boolean

`CASE WHEN vv.id IS NOT NULL THEN 1 ELSE 0 END` vía LEFT JOIN

strongVerification

boolean

`vv.strong_verification` vía LEFT JOIN

Ambos campos provienen del mismo LEFT JOIN a `vendor_verification`.

## Disponibilidad en JWT

Estos campos también quedan embebidos dentro del payload del JWT en cada login.

El frontend puede leerlos directamente del token sin llamar al endpoint.

Campos en JWT:

```
JWT payload → user → vendorVerified
JWT payload → user → strongVerification

```

Aplica a todos los flujos que generan JWT:

Endpoint

POST /auth/login

POST /auth/login/social

POST /auth/autologin

## Archivos a modificar

---

### app/Repository/Auth/AuthRepository.php

Modificar ambos métodos:

- `searchUser()` (línea ~59)


- `getUserById()` (línea ~81)



---

#### Agregar al SELECT

```
CASE WHEN [vv].id IS NOT NULL THEN 1 ELSE 0 END AS vendorVerified,
COALESCE([vv].strong_verification, 0) AS strongVerification

```

---

#### Agregar al FROM

```
LEFT JOIN [LO].[dbo].[vendor_verification] vv ON [u].id = [vv].usuarioID

```

---

### app/Dto/User/UserDto.php

---

#### Nuevas propiedades

```
public $vendorVerified;
public $strongVerification;

```

---

#### Asignación en constructor (después de activeWallet)

```
$this->vendorVerified     = (bool)($user->vendorVerified ?? false);
$this->strongVerification = (bool)($user->strongVerification ?? false);

```

El uso de `?? false` evita errores cuando el `UserDto` se construye sin LEFT JOIN (por ejemplo `registerUserSocial`).
