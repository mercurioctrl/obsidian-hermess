## Descripción

Lo que haremos es agregar el objeto `walletBalance` para traer dos datos fundamentales para operar en el checkout con la billetera

Este objeto tendrá dos campos:

- **available**: Monto total disponible en la billetera del usuario.

- Siempre se obtiene de la base de datos (no de la capa de usuario).


- Se guardará en la tabla `[LO].[dbo].[pedidosCabeceraVendedor]`.




- **usedHere**: Monto que el usuario selecciona usar para el pedido.

- Se guarda también en `[LO].[dbo].[pedidosCabeceraVendedor]`.


- Antes de persistirlo, el sistema debe validar que `usedHere <= available`.


- Además, este monto se **prorrateará automáticamente por cada ítem en **`[LO].[dbo].[pedidosDetalle]`, de forma proporcional al importe del producto.





Ejemplo de prorrateo:

- Pedido con 2 productos de $ 50 cada uno (total: 100).


- Usuario asigna `usedHere = 10`.


- Cada producto recibe `5` en la columna correspondiente a `usedHere` de `[LO].[dbo].[pedidosDetalle]` tal cual se hace con los descuentos.



Si queres podemos agregar descuentos para que veas como funciona esta lógica, suele funcionar el CLIENTEFELIZ25

---

Estos parámetros se almacenaran en `[LO].[dbo].[pedidosCabeceraVendedor]` de modo tal que podamos leerlos y escribirlos al recorrer el checkout

Cada vez que escribamos `available` lo haremos basándonos en nuestros datos de la base de datos y no de la capa de usuario.

Y cada vez que escribamos `usedHere` según lo elegido por el usuario, controlaremos antes de escribir que sea menor o igual a `available`

### Alcance

- Lectura y escritura en `[LO].[dbo].[pedidosCabeceraVendedor]` de ambos parámetros.


- Cálculo de `available` siempre desde DB.


- Validación de `usedHere` antes de guardar.


- Disponibilidad de los datos en el recorrido completo del checkout.



---

### Criterios de aceptación

- `GET /pedidos/checkout/{id}` devuelve el objeto `walletBalance` con `available` y `usedHere`.


- `available` siempre refleja el saldo real del usuario según DB.


- Si el usuario intenta asignar un `usedHere` mayor a `available`, la operación es rechazada con error claro.


- Los valores se persisten en `[LO].[dbo].[pedidosCabeceraVendedor]`.


- El checkout utiliza `usedHere` para el cálculo de pagos, sin permitir inconsistencias.







Para que esta tarea cumpla su proposito de persisitir el saldo utilizado, requiere de una segunda parte que es ajustar el endpoint PATCH /pedidos/checkout . el cual seria el que permite enviar el saldo a utilizar.
