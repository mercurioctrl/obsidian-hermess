## Descripción

Implementar un **proceso automático diario** que:

- **Envíe un email semanal (lunes)** a cada vendedor con su estado actual de objetivo.


- **Al cambiar el mes**, recalculе y actualice `agentes.monthlyTargetAmount` en base a los **3 meses anteriores**, pero **antes** guarde un **snapshot histórico completo** para auditoría y trazabilidad.


- Use la misma tabla histórica para **persistir el mes procesado** y evitar recalcular objetivos más de una vez por mes.



---

### Endpoint

```
PATCH {{API_URL}}/v4/syncUp/salesObjectives
```

- Ejecuta **1 vez por día** (scheduler uptime).


- Idempotente:

- Mailing: solo lunes (y opcionalmente con anti-duplicado).


- Cambio de mes: solo si el mes actual aún no fue procesado.





---

## Persistencia: Tabla de Históricos

### Nueva tabla propuesta

`NB_WEB.dbo.sales_objectives_history` (nombre sugerido; ajustalo a tu convención)

**Propósito**

- Guardar el snapshot mensual por vendedor.


- Registrar el “mes procesado” (fuente de verdad).


- Trazar el cambio de objetivo (previo y nuevo).



**Campos mínimos (propuestos)**

- `id` (PK identity)


- `yearMonth` (ej: `2026-02` o `202602` — elegir uno)


- `sellerId`


- `sellerDescription`


- `amount`


- `targetAmount` *(el objetivo vigente previo al cambio o el vigente al momento del snapshot)*


- `totalSold`


- `totalCost`


- `profit`


- `percentageAchieved`


- `newTargetAmount` *(objetivo recalculado para el nuevo mes; nullable si solo snapshot sin cambio)*


- `createdAt`


- `createdBy` *(opcional: “system”)*


- `runId` *(opcional, para agrupar la corrida diaria)*



**Índices / constraints recomendadas**

- Unique: `(yearMonth, sellerId)` para que haya **un snapshot por vendedor por mes**.


- Index por `yearMonth` para reportes.



---

## Comportamiento esperado

### Email semanal (lunes)

**Condición**

- Solo si el día es lunes.



**Datos**

- Se obtiene desde el recurso existente:
`GET /v1/objectives/totalSale`



**Mail**

- Se envía a `usuarios_nb.UserEmail` según sellerId (mapeo existente).



> Opcional recomendado (si querés evitar duplicados por re-ejecución el mismo lunes):

- registrar en `sales_objectives_history` o una tabla de logs el “mailSentAt” por (yearMonth, sellerId, week) o usar un `sales_objectives_mail_log`.




---

### Cambio de mes (recalcular objetivos) + snapshot obligatorio

**Condición**

- Detectar si el **mes actual** (ej: `2026-02`) **no existe** en `sales_objectives_history` como mes procesado (o no existen registros para ese `yearMonth`).


- Si ya existe, **no recalcular**.



**Orden estricto**

- Calcular métricas actuales por vendedor (misma estructura que `/v1/objectives/totalSale`):

```
{
  "sellerId": 5,
  "sellerDescription": "Patricio Contardi",
  "amount": 18,
  "targetAmount": 113477,
  "totalSold": 4934.41,
  "totalCost": 4425.56,
  "profit": 508.84,
  "percentageAchieved": 4.35
}
```


- **Insertar snapshot en **`sales_objectives_history` para *todos* los vendedores involucrados **antes** de tocar `agentes.monthlyTargetAmount`.


- Recalcular `monthlyAveragePlus20Percent` (3 meses anteriores completos).


- Actualizar `agentes.monthlyTargetAmount` con el nuevo objetivo.


- Actualizar el registro histórico con `newTargetAmount` (o insertarlo ya completo si lo tenés calculado antes).



**Transaccionalidad**

- El paso 2 + 4 debe correr **en una transacción**:

- Si falla la actualización, queda el snapshot (igual sirve para auditoría) **pero** hay que decidir si querés rollback completo.


- Recomendación: **todo en una transacción** para que snapshot y update queden consistentes.
