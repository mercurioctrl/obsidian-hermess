## Descripción

Obtener un listado detallado de los tiempos involucrados en cada etapa del proceso logístico de los pedidos (desde la conversión hasta la entrega), para poder identificar cuellos de botella, evaluar el rendimiento y tomar decisiones operativas.

```
GET {API_URL}/v1/statistics/logisticPerformance?&between=29-06-2025_29-07-2025&sellerId={sellerId}&shippingMethod={shippingMethod}&companyCode={companyCode}&paymentMethod={paymentMethod}&orderStatus={orderStatus}&orderTypeId={orderTypeId}&provinceId={provinceId}
```

Estos esquemas son ostentativos, así que todas las sugerencias son bienvenidas y los cambios deben aplicarse a la historia original.

Usaremos para esto la tabla `[NEW_BYTES].[dbo].[shipping_tracking]` la cual podemos operar usando `DATEDIFF` como muestra el ejemplo o cualquier otro esquema mas performativo.

```
SELECT
    id,
    order_id,
    branch,
    carrier,
    ordered_at,
    ordered_process_at,
    authorized_at,
    build_at,
    dispatched_at,
    label_generated_at,
    delivered_at,

    CASE WHEN ordered_at IS NOT NULL AND ordered_process_at IS NOT NULL
         THEN DATEDIFF(HOUR, ordered_at, ordered_process_at) END AS [tiempo de conversión (h)],

    CASE WHEN ordered_process_at IS NOT NULL AND authorized_at IS NOT NULL
         THEN DATEDIFF(HOUR, ordered_process_at, authorized_at) END AS [tiempo de cobro (h)],

    CASE WHEN authorized_at IS NOT NULL AND build_at IS NOT NULL
         THEN DATEDIFF(HOUR, authorized_at, build_at) END AS [tiempo de armado (h)],

    CASE WHEN build_at IS NOT NULL AND dispatched_at IS NOT NULL
         THEN DATEDIFF(HOUR, build_at, dispatched_at) END AS [tiempo de despacho (h)],

    CASE WHEN label_generated_at IS NOT NULL AND delivered_at IS NOT NULL
         THEN (DATEDIFF(HOUR, label_generated_at, delivered_at)/24) END AS [tiempo de envío (d)]

FROM [NEW_BYTES].[dbo].[shipping_tracking]
WHERE TRY_CAST(ordered_at AS DATETIME) BETWEEN '2025-01-07' AND '2025-29-07'


```

**Filtros obligados:**

- `between`: Rango de fechas en formato `DD-MM-YYYY_DD-MM-YYYY` (obligatorio).




**Filtros opcionales:**

- `sellerId`: ID del vendedor.


- `shippingMethod`: Método de envío.


- `companyCode`: Código de empresa.


- `paymentMethod`: Medio de pago.


- `orderStatus`: Estado del pedido.


- `orderTypeId`: Tipo de pedido.


- `provinceId`: Provincia de destino.



### ✅ **Criterios de Aprobación QA**

- **Estructura y Formato de Respuesta**

- La respuesta es un array de objetos JSON.


- Cada objeto contiene todos los campos especificados:
`orderNumber`, `branchNumber`, `clientDescription`, `clientId`, `orderTypeId`, `observation`, `sellerId`, `seller`, `shippingMethod`, `shippingDescription`, `codePostal`, `provinceId`, `ordered_at`, `ordered_process_at`, `authorized_at`, `build_at`, `dispatched_at`, `label_generated_at`, `delivered_at`, `conversion_time_h`, `payment_time_h`, `build_time_h`, `dispatch_time_h`, `delivery_time_d`.


- **Cálculo de Tiempos**


- `conversion_time_h = ordered_process_at - ordered_at` (en horas)


- `payment_time_h = authorized_at - ordered_process_at` (en horas)


- `build_time_h = build_at - authorized_at` (en horas)


- `dispatch_time_h = dispatched_at - build_at` (en horas)


- `delivery_time_d = delivered_at - dispatched_at` (en días)


- Si alguno de los timestamps involucrados es `null`, el campo calculado correspondiente también debe ser `null`.




- **Filtrado por parámetros**

- El endpoint respeta correctamente los filtros cuando se pasan por querystring y muestra los mismos pedidos que cuando filtramos ordenes:

- `between` (obligatorio): filtra correctamente los pedidos por fecha de creación (`ordered_at`)


- `sellerId`: devuelve solo pedidos del vendedor indicado.


- `shippingMethod`: filtra correctamente por método de envío.


- `companyCode`, `paymentMethod`, `orderStatus`, `orderTypeId`, `provinceId`: funcionan como filtros opcionales si se pasan.






- **Validaciones de parámetros**

- Si no se pasa el parámetro `between`, el endpoint devuelve un error 400 con un mensaje claro.


- Todos los demás filtros son opcionales y no deben generar error si no se incluyen.




- **Casos de borde**

- Si hay pedidos con campos `null` en timestamps, los campos calculados correspondientes también son `null`.




- **Performance**

- El endpoint responde en menos de 2 segundos para rangos de hasta 30 días con al menos 500 pedidos.


- El uso de filtros reduce correctamente la cantidad de resultados y no afecta negativamente la performance.
